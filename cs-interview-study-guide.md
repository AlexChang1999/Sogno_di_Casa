# 軟體工程面試必備：底層原理學習手冊

> 搭配實際案例 + 卡通案例，讓抽象概念變得具體可懂。
> 學習順序：網路全流程 → 資料庫 → Load Balancer/CDN → JWT → Process/Thread

---

## 第一章：網路全流程

### 核心問題：「在瀏覽器輸入網址，到畫面出現，發生了什麼？」

這是面試最常考的題目之一，幾乎每家公司都問過。

---

### 🎬 卡通案例：小新要點外賣

> 小新（你）打開電話，對媽媽說「我要吃大阪燒！」
> 媽媽（DNS）幫你查到大阪燒店的電話號碼（IP）
> 你打電話過去（TCP 連線）
> 店員確認你是合法客人（TLS 握手）
> 你說出你要的品項（HTTP Request）
> 店家做好後送來（HTTP Response）
> 你吃到了（瀏覽器渲染）

---

### 完整流程拆解（共 7 步）

#### 步驟 1：DNS 查詢（翻電話本）

**你輸入** `www.google.com`，電腦不認識這個名字，它只認識 IP（像 `142.250.217.46`）。

DNS（Domain Name System）就是網路的電話本：
- 把「名字」翻譯成「IP 位址」
- 電腦先查本機快取 → 再問路由器 → 再問 ISP → 最後找根域名伺服器

**實際案例：** 你在公司第一次輸入 `github.com`，DNS 查詢需要 50ms；第二次幾乎是 0ms，因為本機有快取記住了。

**面試常問：**
- Q：為什麼需要 DNS？直接記 IP 不行嗎？
- A：IP 難記、會變動、一個服務可能有多個 IP（負載平衡用）。DNS 讓名字穩定、IP 可以隨時更換。

---

#### 步驟 2：TCP 三次握手（確認電話通了）

找到 IP 後，電腦要和伺服器「建立連線」，這叫 **TCP 三次握手**：

```
客戶端 → 伺服器：「你好，我想連你」(SYN)
伺服器 → 客戶端：「好，我聽到了，你也聽到我嗎？」(SYN-ACK)
客戶端 → 伺服器：「聽到了！」(ACK)
```

**卡通案例：** 就像小美打電話給小新：
- 小美：「喂，聽得到嗎？」
- 小新：「聽得到！你呢？」
- 小美：「我也聽得到，開始說吧！」

三次是必要的——少一次，任何一方都無法確認「雙向都通」。

---

#### 步驟 3：TLS 握手（加密信封）

如果網址是 `https://`，在 TCP 建立之後還有一個 TLS 握手，目的是**建立加密通道**，讓中間人無法偷看內容。

**簡化流程：**
1. 瀏覽器：「我支援這些加密方式，你選一個」
2. 伺服器：「好，用這個，這是我的身份憑證（SSL Certificate）」
3. 瀏覽器驗證憑證是真的（由 CA 機構簽發）
4. 雙方交換密鑰，之後的通訊全部加密

**實際案例：** 你在 Sogno di Casa 登入時輸入密碼，TLS 確保密碼在網路上傳輸時是加密的，即使被攔截也看不懂。

---

#### 步驟 4：HTTP Request（點餐）

連線建立後，瀏覽器發送 HTTP Request：

```
GET /index.html HTTP/1.1
Host: www.google.com
Accept: text/html
Cookie: session_id=abc123
```

- **方法**：GET（拿資料）、POST（送資料）、PUT（更新）、DELETE（刪除）
- **Header**：附加資訊，像 Cookie、語言設定
- **Body**：只有 POST/PUT 才有，放要送出的資料

---

#### 步驟 5：伺服器處理（廚房做菜）

伺服器收到請求後：
1. 路由分派：這個網址該由哪個 Controller 處理？
2. 執行邏輯：查資料庫、計算、組裝資料
3. 回傳 Response

**實際案例（Sogno di Casa）：**
你訪問 `/api/products`，Spring Boot 的 `ProductController` 接手，查詢資料庫，回傳 JSON。

---

#### 步驟 6：HTTP Response（送餐）

```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 3421

<html>...</html>
```

**常見狀態碼：**
| 代碼 | 意思 | 卡通比喻 |
|------|------|---------|
| 200 | 成功 | 餐點送達 |
| 301 | 永久轉址 | 搬家了，去新地址 |
| 404 | 找不到 | 菜單上沒這道菜 |
| 500 | 伺服器錯誤 | 廚房著火了 |
| 403 | 沒有權限 | VIP 區不讓進 |

---

#### 步驟 7：瀏覽器渲染（擺盤上桌）

瀏覽器拿到 HTML 後：
1. **解析 HTML** → 建立 DOM 樹（文件結構）
2. **解析 CSS** → 建立 CSSOM（樣式規則）
3. **合併** → Render Tree（哪個元素、放哪裡、什麼樣子）
4. **Layout** → 計算每個元素的位置和大小
5. **Paint** → 真正畫出來

**卡通案例：** 就像建房子：
- HTML = 磚頭（結構）
- CSS = 油漆和裝飾（樣式）
- JavaScript = 住在裡面的人（行為）

---

4. 瀏覽器第二次訪問同一個網站為什麼比較快？（答：快取）

## 第二章：資料庫 Index + Transaction

### 核心問題一：「資料庫 Index 是什麼？為什麼能加速查詢？」

---

### 🎬 卡通案例：圖書館找書

> 圖書館有 100 萬本書，沒有分類系統，你要找《哆啦 A 夢》第 3 集。
> **沒有 Index**：從第一排開始，一本一本翻，直到找到為止（可能翻完整個圖書館）
> **有 Index**：查目錄，目錄說「漫畫區 → 藤子系列 → 第 7 排第 3 格」，直接走過去。

---

### Index 的原理：B-Tree

資料庫的 Index 底層是一棵 **B-Tree（平衡樹）**：

```
              [ 50 ]
            /        \
       [ 25 ]        [ 75 ]
      /      \      /      \
  [10,20] [30,40] [60,70] [80,90]
```

- 每次查詢從根節點開始，比大小決定往左或右走
- N 筆資料只需要 `log₂(N)` 次比較
- 100 萬筆資料，最多只需要 20 次比較！

**沒有 Index** → Full Table Scan（全表掃描），100 萬次比較
**有 Index** → 20 次比較

---

### Index 的代價（沒有免費的午餐）

| | 有 Index | 沒有 Index |
|---|---|---|
| 查詢速度 | 快 | 慢 |
| 寫入速度 | 慢（要更新索引） | 快 |
| 佔用空間 | 多 | 少 |

**實際案例（Sogno di Casa）：**
你的 `Product` 資料表如果有幾萬筆商品，在 `category` 欄位加 Index，`WHERE category = '沙發'` 會從幾秒變幾毫秒。

**什麼時候加 Index？**
- 常被 `WHERE`、`JOIN`、`ORDER BY` 的欄位
- 資料量大（幾千筆以上才有感）
- 不要對每個欄位都加，寫入頻繁的表格加 Index 反而拖慢

---

### 核心問題二：「什麼是 Transaction？ACID 是什麼？」

---

### 🎬 卡通案例：小新幫媽媽轉帳

> 小新的媽媽要把 1000 元從帳戶 A 轉到帳戶 B：
> 1. 從 A 扣 1000 元
> 2. 往 B 加 1000 元
>
> 如果步驟 1 成功，但步驟 2 的時候停電了……
> A 少了 1000 元，B 沒有收到 —— 這 1000 元憑空消失！
>
> **Transaction** 的作用：把步驟 1 和步驟 2 綁在一起。
> 要嘛全部成功，要嘛全部取消（回滾），不存在「做一半」的狀態。

---

### ACID 四大特性

#### A — Atomicity（原子性）
「全部完成」或「全部取消」，不存在中間狀態。
就像原子一樣不可分割。

#### C — Consistency（一致性）
Transaction 前後，資料必須符合所有規則。
例：轉帳前後，A + B 的總金額不能變。

#### I — Isolation（隔離性）
多個 Transaction 同時進行時，互相不干擾。

**卡通案例：** 兩個人同時搶最後一張演唱會票。
- 隔離性確保不會兩個人都以為自己買到了
- 最後只有一個人成功，另一個人得到「售罄」的錯誤

#### D — Durability（持久性）
Transaction 一旦 commit（確認），即使系統崩潰，資料也不會消失。
（透過寫入磁碟的 WAL 日誌實現）

---

### 實際案例（Sogno di Casa）：下訂單

```java
// 偽代碼
@Transactional  // 這個 annotation 就是開啟 Transaction
public Order createOrder(OrderRequest request) {
    // 步驟 1：建立訂單
    Order order = orderRepository.save(new Order(...));
    
    // 步驟 2：扣減庫存
    productService.reduceStock(request.getProductId());
    
    // 如果步驟 2 失敗，步驟 1 也會自動取消（rollback）
    return order;
}
```

如果沒有 `@Transactional`，訂單建立成功但庫存沒扣，就會出現超賣問題。

---

### 面試練習題

1. 如果一個資料表同時被頻繁讀取和寫入，加 Index 好嗎？為什麼？
2. 什麼是 N+1 問題？舉一個例子。
3. 解釋 ACID 中的 Isolation，並說明如果沒有隔離性會發生什麼。

**N+1 問題答案：**
你有 100 個訂單，每個訂單要查一次用戶名稱。
- N+1：先查 1 次取得 100 個訂單，再對每個訂單各查 1 次用戶 = 101 次查詢
- 解法：用 `JOIN` 一次查完，或用 ORM 的 eager loading

## 第三章：Load Balancer + CDN

### 核心問題一：「什麼是 Load Balancer？為什麼需要它？」

---

### 🎬 卡通案例：麥當勞的收銀台

> 中午用餐時間，只有 1 個收銀台，100 個客人排隊，每個人點餐要 2 分鐘。
> 等待時間：最後一個人要等 200 分鐘！
>
> 老闆開了 5 個收銀台，Load Balancer 就是站在門口的店員，
> 看哪個收銀台比較閒，就引導客人去那裡。
> 等待時間：最後一個人只需要等 40 分鐘。

---

### Load Balancer 的工作

```
使用者
  │
  ▼
[Load Balancer]  ← 唯一對外的入口
  ├── 伺服器 A（處理 33% 的請求）
  ├── 伺服器 B（處理 33% 的請求）
  └── 伺服器 C（處理 34% 的請求）
```

**分配策略：**
- **Round Robin**（輪流）：A → B → C → A → B → C...
- **Least Connections**（最少連線）：誰最閒給誰
- **IP Hash**：同一個用戶永遠去同一台伺服器（Session 一致性）

**健康檢查：**
Load Balancer 會定時 ping 每台伺服器，如果伺服器 B 掛了，自動把流量分給 A 和 C，用戶完全感覺不到。

---

### 水平擴展 vs 垂直擴展

| | 垂直擴展 (Scale Up) | 水平擴展 (Scale Out) |
|---|---|---|
| 做法 | 換更強的機器（8核→32核）| 增加更多機器 |
| 成本 | 指數增長 | 線性增長 |
| 上限 | 有（最貴的機器也有極限）| 幾乎無限 |
| 停機時間 | 需要（換機器要停）| 不需要 |

**現代大型服務都用水平擴展 + Load Balancer。**

---

### 核心問題二：「CDN 是什麼？怎麼加速網站？」

---

### 🎬 卡通案例：7-11 的物流

> 統一超商總倉庫在台中，台北的客人要買泡麵：
>
> **沒有 CDN**：每次都從台中倉庫直送台北（1 小時）
> **有 CDN**：台北有個小倉庫（邊緣節點），泡麵存在那裡，直接從台北拿（5 分鐘）
>
> 台中倉庫 = 你的主伺服器
> 台北小倉庫 = CDN 邊緣節點
> 泡麵 = 靜態資源（圖片、CSS、JS）

---

### CDN 的運作原理

```
台灣使用者 → 台灣 CDN 節點（快取有 → 直接回傳）
                           （快取沒有 → 向原始伺服器要，存起來再回傳）

日本使用者 → 日本 CDN 節點
美國使用者 → 美國 CDN 節點
```

**CDN 最適合放什麼？**
- 圖片、影片
- CSS / JavaScript 檔案
- 字型檔案
- 不常變動的 HTML

**不適合放什麼？**
- 需要即時計算的 API 回應
- 登入後的個人資料頁
- 購物車、訂單資訊

**實際案例：** Sogno di Casa 的商品圖片可以放 CDN，讓台灣各地的用戶都能快速載入，同時減輕 Spring Boot 伺服器的壓力。

---

### 面試練習題

1. Load Balancer 和 Reverse Proxy 的差別是什麼？
2. 如果用 Round Robin 分配，但用戶的 Session 存在伺服器 A，下次被分配到伺服器 B，會發生什麼問題？怎麼解決？
3. CDN 的快取更新（Cache Invalidation）是什麼問題？

**第 2 題答案：**
Session 不見了，用戶會被登出。解法有三：
- 用 IP Hash（同個 IP 永遠去同台伺服器）
- Session 存到集中式儲存（Redis），所有伺服器共享
- 用 JWT（無狀態 Token，不需要 Session）← 見第四章

## 第四章：JWT 身份驗證

### 核心問題：「JWT 是什麼？怎麼做身份驗證？和 Session 有什麼差？」

---

### 🎬 卡通案例：迪士尼樂園的手環

> **Session 方式（舊方法）：**
> 入場時，迪士尼在門口登記你的名字和座位號碼（存在他們的名冊裡）。
> 給你一個號碼牌（Session ID）。
> 你每次搭遊樂設施，都把號碼牌給工作人員，工作人員查名冊確認你是誰。
> 問題：如果迪士尼有 100 個園區，每個園區都要有一本名冊，而且要同步！

> **JWT 方式（新方法）：**
> 入場時，迪士尼給你一個特製手環，手環上直接寫著你的名字、票種、有效期限，
> 並且用特殊墨水（數位簽名）防偽。
> 你每次搭遊樂設施，工作人員直接看手環，不需要查名冊。
> 好處：任何一個園區的工作人員都能驗證，不需要中央名冊。

---

### JWT 的結構

JWT 長這樣（用 `.` 分成三段）：
```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyMTIzIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
     Header              Payload                    Signature
```

#### Header（頭）
```json
{
  "alg": "HS256",   // 使用什麼加密演算法
  "typ": "JWT"
}
```

#### Payload（內容）
```json
{
  "sub": "user123",          // 用戶 ID
  "email": "alex@test.com",  // 用戶資訊
  "role": "ADMIN",           // 權限
  "exp": 1735689600          // 過期時間（Unix timestamp）
}
```

#### Signature（防偽簽名）
```
HMACSHA256(
  base64(Header) + "." + base64(Payload),
  你的秘密金鑰          // 只有伺服器知道
)
```

**重要：** Payload 是 Base64 編碼，不是加密，任何人都能解開看內容！
所以 **不要把密碼、信用卡號放進 JWT**。

---

### JWT 登入流程

```
1. 用戶輸入帳號密碼
        │
2. 伺服器驗證正確
        │
3. 伺服器用秘密金鑰簽發 JWT，回傳給用戶
        │
4. 用戶把 JWT 存在 localStorage 或 Cookie
        │
5. 之後每次請求，在 Header 帶上 JWT：
   Authorization: Bearer eyJhbGci...
        │
6. 伺服器收到請求，用秘密金鑰驗證 JWT 是否合法
   - 簽名對嗎？（沒有被篡改）
   - 有沒有過期？
        │
7. 驗證通過 → 處理請求
```

---

### Session vs JWT 比較

| | Session | JWT |
|---|---|---|
| 狀態 | 有狀態（伺服器要記） | 無狀態（資訊在 Token 裡）|
| 儲存位置 | 伺服器記憶體或 Redis | 客戶端（localStorage / Cookie）|
| 水平擴展 | 需要共享 Session（Redis）| 天生支援，任何伺服器都能驗證 |
| 登出 | 簡單（刪 Session）| 複雜（JWT 到期前無法強制失效）|
| 適合場景 | 傳統單伺服器應用 | 微服務、手機 App、多伺服器 |

**實際案例（Sogno di Casa）：**
你的 Spring Boot 後端用 Spring Security + JWT：
- 登入 → 發 JWT
- 每次 API 請求帶 JWT → Spring Security Filter 驗證
- 管理員操作 `/api/admin/**` → 還要確認 JWT 裡的 role 是 `ADMIN`

---

### 面試練習題

1. JWT 的 Payload 可以存密碼嗎？為什麼？
2. 如果用戶的 JWT 被盜了，怎麼辦？
3. JWT 過期時間設多長比較好？有什麼 trade-off？

**第 2、3 題答案：**
- JWT 被盜：因為是無狀態的，到期前無法作廢。解法是使用 Refresh Token 機制 + 短過期時間（15分鐘），一旦發現異常就讓 Refresh Token 失效。
- 過期時間：太長（安全風險），太短（用戶要一直重新登入）。常見做法：Access Token 15 分鐘，Refresh Token 7 天。

## 第五章：Process vs Thread

### 核心問題：「Process 和 Thread 有什麼差？」

---

### 🎬 卡通案例：漢堡店

> **Process（行程）= 一間獨立的漢堡店**
> - 有自己的廚房（記憶體空間）
> - 有自己的食材、工具（資源）
> - 跟別家漢堡店完全獨立，一家著火不影響另一家
>
> **Thread（執行緒）= 同一間店裡的廚師**
> - 共用同一個廚房（記憶體）
> - 可以同時做不同的事（一個烤肉、一個切菜）
> - 廚師 A 不小心打翻鍋，廚師 B 也可能被燙到（共享資源的風險）

---

### Process（行程）

```
Process A                    Process B
┌──────────────────┐         ┌──────────────────┐
│  Code            │         │  Code            │
│  Data（各自的）   │         │  Data（各自的）   │
│  Heap（各自的）   │         │  Heap（各自的）   │
│  Stack           │         │  Stack           │
└──────────────────┘         └──────────────────┘
     記憶體完全隔離
```

- 每個分頁（Tab）在 Chrome 中是獨立的 Process
- 一個 Tab 當掉不影響其他 Tab
- Process 之間要溝通需要特殊機制（IPC）

### Thread（執行緒）

```
Process（一個程式）
┌────────────────────────────────────┐
│  Code（共享）                       │
│  Data（共享）                       │
│  Heap（共享）                       │
│  ┌────────┐ ┌────────┐ ┌────────┐  │
│  │Thread 1│ │Thread 2│ │Thread 3│  │
│  │ Stack  │ │ Stack  │ │ Stack  │  │
│  └────────┘ └────────┘ └────────┘  │
└────────────────────────────────────┘
      記憶體共享，Stack 各自獨立
```

- 一個 Java Spring Boot 伺服器，每個 HTTP 請求通常由一個 Thread 處理
- Thread 之間共享記憶體，溝通很快
- 但共享也帶來「競爭條件（Race Condition）」的風險

---

### 競爭條件（Race Condition）

**卡通案例：**
> 兩個廚師（Thread A 和 Thread B）都看到剩下 1 顆雞蛋。
> A 說：「還有 1 顆，我來用」同時 B 說：「還有 1 顆，我來用」
> 結果兩個人都去拿，但只有 1 顆，其中一個拿到空的。

**程式碼範例：**
```java
// 危險！兩個 Thread 同時執行這段程式碼
int count = 0;

// Thread A 和 Thread B 同時執行：
count = count + 1;

// 預期結果：2
// 實際可能：1（因為兩個 Thread 同時讀到 0，各自加 1，寫回 1）
```

**解法：加鎖（synchronized / Lock）**
```java
// 安全，同時只能有一個 Thread 進入
synchronized void increment() {
    count = count + 1;
}
```

---

### Deadlock（死鎖）

**卡通案例：**
> 小新要番茄醬和醬油才能做菜
> 小葵要醬油和番茄醬才能做菜
>
> 小新拿到番茄醬，在等醬油
> 小葵拿到醬油，在等番茄醬
>
> 兩個人互等，永遠做不了菜 → Deadlock

**在程式中：**
- Thread A 持有鎖 1，等待鎖 2
- Thread B 持有鎖 2，等待鎖 1
- 互相等待，系統卡死

**避免方法：** 所有 Thread 都按照相同的順序取鎖（例如：永遠先拿鎖 1 再拿鎖 2）

---

### 實際應用：Spring Boot 的 Thread 模型

```
HTTP 請求進來
      │
Spring Boot 的 Thread Pool（預設 200 個 Thread）
      │
從 Pool 取一個空閒的 Thread 處理這個請求
      │
處理完畢，Thread 歸還給 Pool，等待下一個請求
```

這就是為什麼 Spring Boot 能同時處理幾百個請求——不是因為有幾百個 CPU，而是用 Thread 讓 CPU 快速切換。

### 場景：你在 Sogno di Casa 購買一件家具

```
1. 你在瀏覽器輸入 sogno-di-casa.com

   → TLS 加密

   → API 請求打到 Load Balancer（第三章）
   → Load Balancer 分配給其中一台 Spring Boot 伺服器

3. 你點擊「加入購物車」
   → 瀏覽器帶著 JWT 發送請求（第四章）
   → Spring Security 驗證 JWT，確認你是誰

4. 你點擊「結帳」
   → 後端開啟 Transaction（第二章）
   → 建立訂單 + 扣除庫存（原子操作，不能只做一半）
   → 資料庫查詢用 Index 加速（第二章）

5. 伺服器處理這個請求
   → 被分配到一個 Thread（第五章）
   → Thread 完成後歸還 Thread Pool

6. 回傳結果給瀏覽器
   → HTTP Response 200 OK
   → 瀏覽器渲染「訂單成立」畫面（第一章）
```
